\section{Introduction}

The goal of this project was to compare three stochastic optimization methods on
the same convex machine learning problem. The methods were stochastic gradient
descent (SGD), stochastic variance-reduced gradient (SVRG), and SAGA. All three
methods were tested on L2-regularized logistic regression using the breast
cancer dataset from \texttt{sklearn}.

This is a useful comparison because the three methods make different tradeoffs.
SGD is simple and cheap per iteration, but its gradient estimates can stay noisy.
SVRG reduces this noise by occasionally computing a full gradient at a snapshot
point. SAGA also reduces variance, but it does this by storing past gradients
for all samples. Because of this, the main question is not only which method
reaches the lowest objective value, but also how much extra computation and
memory that improvement costs.

The main result was that SVRG reached the lowest final objective value in the
locked comparison, but it used about three times as many gradient evaluations as
SGD. SAGA was competitive when tuned well, especially at learning rate 0.03, but
at the locked learning rate 0.1 it was less stable than SVRG. SGD was the
simplest and usually fastest method in wall-clock time, but its final loss was
higher and its trajectory had more noise.

\section{Problem Setup}

The optimization problem was binary logistic regression with L2 regularization.
For data points $(x_i, y_i)$ with labels $y_i \in \{-1, +1\}$, the objective was
\[
F(w,b)
= \frac{1}{n} \sum_{i=1}^{n}
\log\left(1 + \exp\left(-y_i(w^\top x_i + b)\right)\right)
+ \frac{\lambda}{2}\|w\|_2^2 .
\]
The bias term $b$ was not regularized.

The dataset was \texttt{sklearn.datasets.load\_breast\_cancer}. It has 569
examples and 30 features. The labels were converted from the original
\texttt{\{0,1\}} format to $\{-1,+1\}$. The features were standardized using
\texttt{StandardScaler}. The class counts after conversion were 212 examples
with label $-1$ and 357 examples with label $+1$.

All runs used
\[
\lambda = 10^{-3}.
\]
The starting point was the zero vector for both $w$ and $b$. At this point the
objective value was 0.69314718. I also checked the implementation of the
gradient using finite differences. The maximum finite-difference error reported
by the project code was $7.173 \times 10^{-11}$, and the mean of the
per-sample gradients matched the full gradient up to numerical precision.

\section{Methods}

\subsection{SGD}

SGD used one randomly sampled data point per update. For a sampled index $i_t$,
the update was
\[
\theta_{t+1} = \theta_t - \eta \nabla f_{i_t}(\theta_t),
\]
where $\theta = (w,b)$ and $f_i$ is the single-example logistic loss plus the
regularization contribution. SGD only needs the current parameter vector and one
stochastic gradient, so it has the smallest memory cost. The drawback is that
the gradient estimate can have high variance, especially near the optimum.

\subsection{SVRG}

SVRG keeps a snapshot point $\tilde{\theta}$ and computes the full gradient
$\nabla F(\tilde{\theta})$ at that point. During the inner loop, it uses the
variance-reduced estimator
\[
v_t =
\nabla f_{i_t}(\theta_t)
- \nabla f_{i_t}(\tilde{\theta})
+ \nabla F(\tilde{\theta}).
\]
Then it updates $\theta_{t+1} = \theta_t - \eta v_t$.

In this project, the inner loop length was set to one epoch, meaning 569 inner
updates per snapshot. This makes the method easy to compare against SGD, but it
also means SVRG pays for a full pass over the data at the start of every epoch.
So SVRG has low memory overhead, but higher gradient-evaluation cost.

\subsection{SAGA}

SAGA stores a table of past per-sample gradients. At each step it samples one
index, computes a new gradient for that sample, and combines it with the stored
table average. In the implementation, the gradient table stored the
unregularized per-sample gradients, and the L2 gradient was added fresh at the
current iterate. This avoids making the stored gradients depend on an old value
of the regularization term.

SAGA has a lower gradient-evaluation count than SVRG in these experiments
because it does one new stochastic gradient per update after initializing the
gradient table. The main cost is memory: it stores one gradient vector for every
training example. On this dataset that memory is still small, but it would grow
linearly with both the number of examples and the number of features.

\section{Experimental Setup}

All methods used the same dataset, preprocessing, initialization, and objective.
Each main run used 30 epochs. One epoch means 569 stochastic update steps for
SGD and SAGA. For SVRG, one epoch means one snapshot full-gradient computation
and 569 inner-loop updates.

The project tracked three main metrics:
\begin{itemize}
    \item objective value,
    \item number of gradient evaluations,
    \item wall-clock runtime in seconds.
\end{itemize}

The learning-rate sweeps used seed 0. The tested learning rates were
\[
\text{SGD: } \{0.001, 0.003, 0.01, 0.03, 0.1\},
\]
\[
\text{SVRG and SAGA: } \{0.005, 0.01, 0.03, 0.05, 0.1, 0.2\}.
\]
After the sweep, the locked learning rates used in the main comparison were
0.03 for SGD, 0.1 for SVRG, and 0.1 for SAGA.

Table~\ref{tab:tradeoff} summarizes the practical tradeoffs. The memory numbers
are based on the active optimizer state in the implementation, using 64-bit

does not count the dataset itself, since all methods use the same data matrix.

\begin{table}[H]
\centering
\caption{Compute and memory tradeoffs for the locked setup.}
\label{tab:tradeoff}
\small
\resizebox{\textwidth}{!}{%
\begin{tabular}{lrrrrl}
\toprule
Method & Final loss & Grad evals & Runtime (s) & State memory & Main tradeoff \\
\midrule
SGD  & 0.0641138605 & 17070 & 0.1774 & about 0.5 KB

SVRG & 0.0598532343 & 51210 & 0.3122 & about 1.2 KB

SAGA & 0.0627392703 & 17639 & 0.2262 & about 138.5 KB
& more memory, fewer evals than SVRG \\
\bottomrule
\end{tabular}
}
\end{table}

For this small dataset, the memory cost of SAGA is not a real problem. Its
gradient table is only about 138.5 KB. But the scaling matters. If the dataset
had millions of examples or a much larger feature dimension, SAGA's $O(nd)$
memory could become expensive. SGD and SVRG only need $O(d)$ optimizer state, so
they are much easier to scale in memory.

In terms of convergence quality, SVRG was the strongest method in the locked
comparison. It reached the lowest loss and had the smoothest final behavior.
The cost is that it used 51210 gradient evaluations, compared with 17070 for SGD
and 17639 for SAGA. So SVRG is attractive when gradient evaluations are cheap
enough and we care about getting a slightly lower objective value.

SAGA is most worth its memory cost when the gradient table fits comfortably in
memory and when a well-tuned learning rate is used. In the sweep, SAGA with
learning rate 0.03 reached 0.0603328688, which was close to SVRG's 0.0598532343,
while using about one third as many gradient evaluations. That is the strongest
case for SAGA in this project. The weaker case is the locked learning rate 0.1,
where SAGA was still reasonable but not as good as its tuned result.

SGD is still useful as a baseline. It was easy to implement, used the least
memory, and had the fastest locked runtime. But it did not reach the same final
objective value as the variance-reduced methods. In this experiment, SGD is the
best choice if simplicity and runtime are more important than squeezing out the
lowest objective value.

\section{Conclusion}

This project compared SGD, SVRG, and SAGA on the same convex logistic regression
problem. The experiment was intentionally kept simple: same dataset, same
preprocessing, same initialization, same number of epochs, and the same
objective for all methods.

The main takeaway is that variance reduction helped. SVRG reached the lowest
final loss in the locked comparison and was also very stable across the seed
check. Its downside was the extra compute from repeated full-gradient
calculations. SAGA was competitive when tuned properly, especially at learning
rate 0.03, where it came close to SVRG while using far fewer gradient
evaluations. Its main downside is the gradient table memory, which is small here
but can matter for larger datasets.

Overall, I would choose SGD for the simplest and fastest baseline, SVRG when I
want the most reliable final objective value and can afford more gradient
evaluations, and SAGA when the gradient table fits in memory and the learning
rate has been tuned carefully.
